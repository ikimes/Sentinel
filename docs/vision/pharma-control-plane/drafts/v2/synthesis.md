## Current v2 Strategic Core

Sentinel should enter as a narrow, evidence-grade overlay for one recurring cross-system regulated follow-up matter, earn trust through a daily operator workspace and minimal-footprint proof, and expand only if the same matter-centered coordination model survives without turning into bespoke services.

## What Sentinel Should Copy

- Copy entry through one painful operational thread, not through broad category language.
- Copy evidence-grade trust and reconstructability as part of the product core.
- Copy minimal-footprint time to value.
- Copy operator-first workflow surfaces rather than dashboard-first or graph-first surfaces.
- Copy coexistence with incumbent systems and expansion by repeated matter pattern.
- Copy the commercial entry behavior of a smaller control surface only when it still points back to the deeper matter-workspace product.

## What Sentinel Should Avoid

- Avoid platform-first or suite-replacement positioning.
- Avoid any product shape that can be mistaken for generic workflow, dashboards, graphing, or AI summaries.
- Avoid first proofs that need many integrations or a broad rules engine before value appears.
- Avoid customer-specific model sprawl and services drift.
- Avoid weakly qualified customers whose pain is visible but not concentrated.
- Avoid preserving strategic elegance if it slows down proof of the first useful workflow.

## What Sentinel Must Uniquely Prove

- Prove that the first matter type creates real buying pressure, not just conceptual agreement.
- Prove that a real overlay buying path exists with a credible economic owner.
- Prove that Sentinel's matter model and workspace outperform internal build on coherence and repeatability.
- Prove that one source system, one downstream target, and a small user group are enough to show value.
- Prove that the workspace answers owner, clock, blocker, downstream-state, and decision-basis questions better than incumbent combinations.
- Prove that any thinner commercial wrapper still compounds into the same deeper product.

## Interdependencies

- The `copy` set only works if the `avoid` set keeps Sentinel from collapsing into adjacent categories.
- The `avoid` set only matters if the `uniquely prove` set turns those boundaries into earned evidence rather than doctrine alone.
- The `uniquely prove` set depends on the `copy` set to define what good looks like and on the `avoid` set to define what false positives look like.
- Together, the three axes point to the same center: Sentinel should be productized as a matter workspace, bought as a thin overlay, and validated through one recurring workflow that beats internal build and incumbent stitching.

## What This Means For Post-Discovery Work

- Buyer validation should focus first on overlay appetite, named budget path, and recurrence of the first matter type.
- MVP proof should prioritize one source system, one downstream target, replayable scenarios, and the five core operator questions.
- UI work should stay centered on the matter workspace, path, systems board, and provenance-backed context window, with graph mode kept secondary.
- System design should protect the stable matter model and resist early pressure to become a broad regulatory-content engine or generic workflow layer.
