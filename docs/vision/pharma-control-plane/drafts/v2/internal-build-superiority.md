## Purpose

This workstream isolates the strongest substitute for Sentinel:

internal build on top of incumbent systems, ServiceNow-like workflow layers, dashboards, and custom integrations.

The question is not whether buyers can build something.

The question is whether Sentinel can prove that buying it is more credible than stitching the same answer together themselves.

## Current Read

Internal build is dangerous because it is:

- politically easy to justify
- able to reuse existing teams and budget
- good enough for many local workflows
- hard to dislodge when the pain is only moderate

Sentinel only wins if it proves superiority on product coherence and repeatability, not on abstract elegance.

## Where Internal Build Is Strong

Internal build is genuinely strong when:

- the workflow is narrow and local
- the buyer already has strong workflow teams
- the matter mostly stays inside one incumbent system
- leadership only wants visibility or light exception handling
- the organization prefers local flexibility over reusable product shape

This is why the weaker hypothetical partner does not qualify.

## Where Internal Build Starts To Break

Internal build weakens when:

- the workflow crosses several systems and teams repeatedly
- ownership and due-state must stay coherent over time
- evidence reconstruction becomes operationally important
- each new workflow starts to require more custom logic and more custom views
- the result drifts toward tickets, reports, and one-off integrations

That is where Sentinel's deeper product thesis matters.

## What Sentinel Must Beat

Sentinel does not need to beat internal build on raw flexibility.

It needs to beat internal build on these dimensions:

### 1. Coherent matter model

Sentinel should give buyers a stable object of work rather than one more local workflow project.

### 2. Evidence-grade provenance

Sentinel should preserve evidence, decisions, and downstream state by design rather than by convention.

### 3. Operator workspace quality

Sentinel should answer owner, clock, blocker, downstream-state, and decision-basis questions better than tickets, dashboards, and stitched views.

### 4. Repeatability across adjacent workflows

Sentinel should show that the same model can survive another similar workflow without becoming custom consulting.

### 5. Time to proof

Sentinel should show value with a smaller and more coherent first footprint than an internal project would typically require.

## Weak Arguments Sentinel Should Not Use

- "internal teams cannot build this"
- "our graph is better"
- "our AI is smarter"
- "integration is hard"

Those are weak because they are either not credible or not central.

## Strongest Buy Argument Against Internal Build

The strongest current argument is:

Sentinel productizes the matter model, evidence trail, and operator workspace that internal teams would otherwise rebuild inconsistently inside ticketing, dashboards, and one-off workflow layers.

That argument is strongest when the buyer already knows the problem crosses systems and keeps recurring.

## What Sentinel Still Has To Demonstrate

### 1. The model really stays stable

If each buyer or workflow needs new core entities, internal build becomes more rational again.

### 2. The workspace is a better daily tool, not just a cleaner concept

If operators would still rather work from incumbent screens plus reports, Sentinel loses.

### 3. The first implementation is lighter than a local custom project

If Sentinel needs a major rollout just to prove itself, internal build keeps its edge.

### 4. The product compounds over time

If the second adjacent workflow is no easier than the first, Sentinel looks less like product and more like packaged consulting.

## Strongest Questions To Test Next

- what would an internal team likely build first for this workflow
- how long would that take before operators actually trust it
- where would provenance, evidence, and downstream-state likely drift
- what parts of Sentinel are genuinely reusable versus locally configurable
- does the second workflow become easier because Sentinel already exists

## Evidence Sentinel Still Needs

- one scenario where Sentinel answers the five core operator questions more cleanly than ticketing plus reporting
- one scenario where the same matter model survives a second closely related workflow
- one proof that the first delivery footprint is lighter than a credible internal alternative
- one prospect statement close to:
  - "we could build this, but buying Sentinel is more credible than stitching it together again"

## Working Decision Rule

Treat superiority over internal build as earned only when Sentinel can show all three:

- stronger product coherence
- stronger repeatability
- faster path to useful operational proof

If Sentinel only looks nicer, more strategic, or more elegant, internal build remains the stronger substitute.
