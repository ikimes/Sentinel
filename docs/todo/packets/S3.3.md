## Interfaces Affected

Do not add or change backend API contracts.

Continue consuming only the current matter-read surfaces:

- `GET /api/compliance/matters/queue`
- `GET /api/compliance/matters/{scenarioId}/overview`
- `GET /api/compliance/matters/{scenarioId}/timeline`

## Required Product Shape

Keep the operator-first shell structure:

- queue
- matter workspace
- timeline
- context window

Do not add graph mode, new routes, or new product capability in this slice.

## Scope Boundary

This slice is primarily:

- content refinement
- hierarchy refinement
- wording refinement
- light visual hierarchy tuning only where it directly supports comprehension

This slice is not the full styling system pass.

## Required Refinement Focus

Address these inspection conclusions directly:

1. the page needs a more natural flow from triage into understanding
2. the matter workspace is too congested
3. scenario differences are not legible enough at a glance
4. some labels sound internal or implementation-driven
5. `matter` may need softening or brief in-product grounding, even if the domain term remains

## Required Behaviors

After this slice:

1. the selected matter should be more obvious visually
2. the matter workspace should read more clearly in sections such as:
   - current state
   - why in focus
   - next move
3. the context window title and wording should be more operator-facing
4. queue views should better communicate why an operator would use them
5. timeline time presentation should be more human-readable than raw UTC formatting

## Preferred Implementation Shape

- keep one primary page
- prefer copy and layout changes over new components unless extraction clearly improves clarity
- allow small CSS changes that improve information hierarchy
- keep the grayscale-first direction, but do not implement the full theme system yet

## Guardrails

- do not widen feature scope
- do not add real actions, forms, or write flows
- do not add auth work
- do not add graph mode
- do not introduce frontend-only workflow semantics
- do not implement full light/dark mode in this slice

## Implementation Steps

1. Refine the hero panel so it explains the page job more clearly and takes less screen attention.
2. Improve queue-view language and supporting copy so filter intent is clearer.
3. Improve queue-card scannability and selected-state clarity.
4. Restructure the matter workspace to reduce congestion and better separate state, risk, and next move.
5. Rename and refine the context window so it feels operator-facing rather than implementation-facing.
6. Improve timeline readability, especially timestamp presentation and event emphasis.
7. Apply only the minimum visual hierarchy changes needed to support the content goals.
8. Update the inspection doc only if you resolve an open question or narrow a recommendation.
9. Update the execution board only enough to reflect `S3.3` progress.

## Acceptance Criteria

- the shell remains operator-first
- the page reads more naturally from queue to workspace to timeline to context
- the matter workspace is meaningfully less congested
- queue differences are easier to understand at a glance
- at least one operator-facing framing improvement is made around `matter` or the surrounding copy
- timeline times are more human-readable
- no backend or API changes are introduced
