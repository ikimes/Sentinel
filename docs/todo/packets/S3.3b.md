## Goal

Do a second semantic refinement pass on the Blazor shell so the queue signals, surfaced reasons, and matter distinctions are trustworthy and operator-facing before the full styling-system pass.

If those issues remain, `S3.4` will polish ambiguity instead of clarifying it.

## Required Focus

This slice is intentionally narrow.

It should fix only the remaining semantic clarity issues discovered after `S3.3`:

1. queue signals should match the actual state more faithfully
2. `Why it surfaced` should read like operator language, not internal codes
3. queue-card differences should be easier to understand immediately
4. matter-level copy should better communicate consequence and action relevance
5. do not broaden into the full styling/token/light-dark pass yet

## Locked Semantic Targets

Use the current three Stage 4 proof scenarios as explicit targets for the second-pass wording.

The exact final copy may vary slightly, but the semantic outcome must be this sharp:

### `de-affiliate-delay-path`

- queue signal should communicate deadline pressure or missing evidence, not a generic status
- surfaced reason should communicate that the matter is still waiting on Germany affiliate evidence while the review deadline remains active
- selected-matter copy should make it obvious this is the active-deadline / evidence-gap scenario

### `de-affiliate-downstream-retry`

- queue signal should communicate blocked or retried downstream handoff, not deadline pressure
- surfaced reason should communicate that Germany affiliate work is complete but downstream acknowledgement is still not settled after retry
- selected-matter copy should make it obvious this is the downstream-friction / retry scenario

### `de-affiliate-happy-path`

- queue signal should communicate resolved or acknowledged state, not deadline pressure
- surfaced reason should communicate that the matter remains visible only because it is in the full queue view, not because it currently needs intervention
- selected-matter copy should make it obvious this is the acknowledged / settled scenario

## Required Behaviors

After this slice:

1. queue-signal labels should be more trustworthy than `S3.3`
2. surfaced reasons should be readable in operator language
3. the three current proof scenarios should feel more distinct in the queue and selected-matter area
4. summary copy should better explain consequence, not just state
5. any remaining hero/meta redundancy should be reduced if it does not earn its space

More concretely:

- the delay-path matter should no longer look semantically similar to the downstream-retry matter
- the happy-path matter should no longer surface a misleading urgency cue
- a reviewer should be able to tell, within one queue scan, which item is:
  - deadline-sensitive
  - blocked on downstream follow-through
  - already acknowledged / settled

2. Replace raw `Why it surfaced` values with operator-facing phrasing.
3. Tighten queue-card summaries so they convey consequence more clearly.
4. Improve selected-matter subtitle or surrounding copy so the chosen scenario is more distinguishable.
5. Revisit hero meta or other minor repeated copy only if it directly improves clarity.
6. Update the inspection doc only if you resolve an open semantic issue.
7. Update the execution board only enough to reflect `S3.3b` progress.
