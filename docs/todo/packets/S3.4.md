## Required Styling Direction

Keep the current grayscale-first instinct, but refine it into an intentional system.

The shell should gain:

- clearer visual hierarchy
- a small semantic color system
- explicit light mode and dark mode support
- better distinction between:
  - static panels
  - dynamic panels
  - interactable surfaces
  - status/bubble elements

## Required Theme Support

This slice must include both:

- light mode
- dark mode

Theme support should be implemented using reusable design tokens or CSS variables rather than ad hoc overrides.

Theme behavior must be explicit, not implicit only.

Required theme behavior:

- support system theme detection
- provide an in-app theme toggle so the user can switch between light and dark mode directly from the shell

If a worker believes the toggle should not be added in this slice, they should stop and hand back the decision rather than silently implementing system-only theme behavior.

## Required Semantic Color Roles

Introduce a restrained set of semantic accents for at least:

- urgency / active clock
- blocked
- resolved / acknowledged
- selected / current focus

Do not introduce a broad or noisy color palette.

## Preferred Implementation Shape

- use CSS variables for theme tokens
- keep the shell visually restrained and product-like
- improve panel depth, borders, surface separation, and status affordances
- let visual styling reinforce the content hierarchy established in `S3.3`

## Current Shell Role Map

Use the current shell structure as the styling anchor. Do not reinterpret roles broadly.

- `Static`
  - hero definition and explanatory copy
  - operator brief footnote
- `Dynamic`
  - selected matter workspace
  - timeline
  - operator brief body
- `Interactable`
  - queue items
  - queue lenses
- `Status/Bubble`
  - queue signals
  - downstream state pills
  - selected-state chips
  - scenario tags

The styling pass should reinforce these roles rather than invent a new surface taxonomy.

## Guardrails

- do not widen product scope
- do not add graph mode
- do not add JS interop
- do not replace the shell structure
- do not let color overpower readability
- do not regress mobile usability

## Implementation Steps

1. Establish theme tokens for light and dark mode.
2. Refine the shell surfaces so panel roles are visually clearer.
3. Improve bubble/status treatment with restrained semantic color.
4. Improve selected-state treatment for queue items.
5. Improve workspace emphasis for blocker, urgency, and next move.
6. Ensure the shell remains strong on mobile.
7. Update the inspection doc only if the styling pass resolves a current open question.
8. Update the execution board only enough to reflect `S3.4` progress.

## Acceptance Criteria

- the shell supports both light and dark mode
- the shell includes an in-app theme toggle
- the grayscale-first direction remains intact but more intentional
- semantic color is present and restrained
- panel role differences are easier to perceive
- selected matter state is visually clearer
- mobile usability remains intact
- no backend or API changes are introduced
