## Interfaces Affected

Add one narrow internal-proof HTTP surface:

- `GET /api/compliance/matters/queue`

This slice should treat the three locked replay scenario ids as the proof-matter set:

- `de-affiliate-happy-path`
- `de-affiliate-delay-path`
- `de-affiliate-downstream-retry`

Add one queue response contract and one queue item contract.

Required query support:

- optional `view` parameter with these supported values:
  - `all`
  - `active-clock`
  - `blocked-downstream`
  - `missing-evidence`
  - `missing-owner`

Default behavior:

- `view=all` when omitted or empty

Required queue item fields:

- `scenarioId`
- `matterName`
- `jurisdiction`
- `owner`
- `activeClock`
- `blocker`
- `downstreamState`
- `downstreamAction`
- `decisionBasis`
- `queueReason`

## Preferred Implementation Shape

- keep the queue backed directly by `ComplianceMatterReplayHarness`
- reuse the existing overview mapping logic where helpful instead of duplicating matter-field derivation
- keep sorting deterministic and simple
- do not introduce pagination, search, generalized filtering, or dashboard abstractions

## Required Queue Semantics

Use these narrow rules:

- `all`
  - return all three proof matters in deterministic order
- `active-clock`
  - include matters where the replay clock is active
- `blocked-downstream`
  - include matters where downstream action is still pending because of technical or coordination state
- `missing-evidence`
  - include matters where the blocker is missing affiliate review evidence
- `missing-owner`
  - include matters with no owner; for the current locked scenarios this may legitimately return an empty list

Required deterministic ordering:

1. active clock matters first
2. then blocked downstream matters
3. then remaining matters by `scenarioId` ascending

## Implementation Steps

1. Add queue response and queue item response types in the compliance contracts area.
2. Add one read-service method that projects the locked replay scenarios into queue items.
3. Add the `view` filter using only the supported values above.
4. Map `GET /api/compliance/matters/queue` in the existing endpoint module.
5. Treat unsupported `view` values as `400 Bad Request`.
6. Add focused tests for:
   - default `all` queue returns all proof matters in deterministic order
   - `active-clock` isolates the delay-path matter
   - `blocked-downstream` includes the retry-path matter
   - `missing-evidence` includes the delay-path matter
   - `missing-owner` returns an empty list for the current locked scenarios
   - unsupported `view` returns `400`
7. Update the execution board only to reflect `S2.5` progress.

## Acceptance Criteria

- one narrow queue/exception endpoint exists and is backed by the replay harness
- the proof matters can be triaged from one small operational read surface
- the queue exposes the key exceptions without becoming a generic reporting layer
- unsupported `view` values return `400`
- the slice does not add dashboards, search, pagination, UI work, or generalized workflow infrastructure
