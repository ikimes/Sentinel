## Interfaces Affected

This slice should consume existing HTTP surfaces only:

- `GET /api/compliance/matters/queue`
- `GET /api/compliance/matters/{scenarioId}/overview`
- `GET /api/compliance/matters/{scenarioId}/timeline`

Do not add or change API contracts in this slice.

## Required Product Shape

Build the first operator shell with these default surfaces:

- matter workspace
- queue
- timeline
- context window

The graph/expert mode is explicitly out of scope for this slice.

## Preferred Implementation Shape

- add a new Blazor project at `Sentinel.Web`
- add it to the solution
- wire it into Aspire in `Sentinel.AppHost/Program.cs`
- keep the UI shell intentionally narrow and data-driven from the existing API
- prefer a single primary page or layout that proves the shell, rather than broad route expansion

## Required UI Behaviors

Implement at least these flows:

1. the app loads the matter queue on first render
2. the first queue item is selected by default when data exists
3. selecting a queue item loads:
   - the matter overview
   - the matter timeline
4. the main screen shows:
   - owner
   - active clock
   - blocker
   - downstream state
   - decision basis
5. the timeline is visible and readable without opening raw infrastructure logs
6. the context window explains the currently selected matter using existing backend fields only

## Context Window Rule

Do not invent a new AI service in this slice.

The context window should be a deterministic UI composition from current read-model fields, for example:

- what happened
- why it matters
- next expected action

This is a presentation-layer composition only.

## Design Guardrails

- keep the workspace operator-first
- do not make the graph or systems board the default view
- do not add auth or actor-seam work yet
- do not add JS interop yet unless it is absolutely required for basic Blazor shell behavior
- do not invent client-side workflow semantics that are not present in the backend contracts

## Implementation Steps

1. Create the new Blazor project in `Sentinel.Web`.
2. Add it to `Sentinel.sln`.
3. Wire it into `Sentinel.AppHost/Program.cs`.
4. Add one typed client/service layer in the Blazor project for the three existing matter endpoints.
5. Build the first workspace shell with queue, overview, timeline, and context composition.
6. Make the default selected matter deterministic.
7. Keep styling intentional but narrow; prioritize clarity of owner, clock, blocker, downstream state, and decision basis.
8. Update the execution board only to reflect `S3.1` progress.

## Acceptance Criteria

- a new Blazor frontend project exists in the solution
- the app consumes the existing matter queue, overview, and timeline endpoints
- the default experience is matter workspace first
- queue, timeline, and context are all visible in the first shell
- no backend API contracts are changed
- the slice does not introduce graph mode, auth work, or frontend-only workflow drift
