# Slice Packet: S3.2

## Goal

Prepare the internal actor/auth seam so the Blazor workspace no longer depends directly on ad hoc trusted actor-header handling.

## Why Now

`S3.1` added the first Blazor matter workspace shell. The next step is to wrap the temporary trusted-header model behind one UI-side identity provider and one API-side resolver so future submit and action flows can evolve toward real authentication without changing product contracts.

## Interfaces Affected

Do not change public API contracts.

Preserve current actor persistence fields and semantics:

- `actorId`
- `actorType`
- `actorDisplayName`

Preserve current trusted internal header names underneath the new seam:

- `X-Actor-Id`
- `X-Actor-Type`
- `X-Actor-Display-Name`

## Required Seam Shape

Implement one abstraction on each side:

- UI side: one internal actor identity provider used by the Blazor app
- API side: one actor resolver used by compliance acceptance logic

The purpose is not real auth yet. The purpose is to stop the app and API from depending directly on scattered header parsing and header-setting behavior.

## Preferred Implementation Shape

UI side:

- add one `IInternalActorProvider` or equivalent in `Sentinel.Web`
- add one delegating handler or equivalent request-layer seam that applies actor headers centrally
- wire the matter workspace client through that seam instead of raw `HttpClient` header handling

API side:

- extract actor resolution behind one interface or service used by compliance acceptance
- keep trusted-header parsing as the current implementation behind that interface
- register the seam centrally in `Program.cs`

## Guardrails

- do not add real login, cookies, JWTs, OIDC, or SSO
- do not change the Stage 4 matter endpoints
- do not change actor field names persisted in the audit/read model
- do not broaden into permissions or authorization rules
- do not invent frontend-only actor state that can drift from what the API receives

## Required Behavior

After this slice:

1. the Blazor app should have one obvious place where internal actor identity is defined
2. outbound calls from the Blazor shell should apply actor headers through one central seam
3. the API should resolve actor metadata through one central seam
4. current audit/read-model behavior must remain intact
5. the repo should be easier to evolve toward real auth later without changing the product-facing contracts

## Implementation Steps

1. Add the UI-side internal actor provider abstraction in `Sentinel.Web`.
2. Add one request-layer seam that applies actor headers centrally to the workspace client.
3. Refactor the Blazor registration to use that seam.
4. Add the API-side actor resolver abstraction around the current trusted-header parsing.
5. Refactor compliance acceptance to use the resolver instead of direct static parsing.
6. Add or update focused tests for:
   - header normalization still works
   - actor resolver returns the same persisted values as before
   - the UI seam applies the expected headers in one place, if feasible without broad test setup
7. Update docs/board wording only enough to reflect `S3.2` progress.

## Acceptance Criteria

- one UI-side internal actor provider exists
- one API-side actor resolver exists
- the current trusted-header model is preserved behind those seams
- current actor persistence semantics still hold
- no public API contract changes are introduced
- the slice clearly improves future replaceability with real auth
