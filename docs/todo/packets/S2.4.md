## Interfaces Affected

Add one narrow internal-proof HTTP surface:

- `GET /api/compliance/matters/{scenarioId}/timeline`

Reuse the locked replay scenario ids as the matter identifiers for this proof slice:

- `de-affiliate-happy-path`
- `de-affiliate-delay-path`
- `de-affiliate-downstream-retry`

Add one timeline response contract and one event response type.

Required response fields:

- `scenarioId`
- `matterName`
- `jurisdiction`
- `events`

Required event fields:

- `timestampUtc`
- `eventType`
- `summary`
- `owner`
- `blocker`
- `evidence`
- `downstreamState`

## Preferred Implementation Shape

- keep the timeline backed directly by `ComplianceMatterReplayHarness`
- add one narrow read-service method next to the overview method
- keep the route and contract in the existing compliance feature area
- preserve deterministic ordering exactly as emitted by the replay harness

## Implementation Steps

1. Add a matter timeline response type and event response type in the compliance contracts area.
2. Extend the read service with one method that maps replay output into the timeline shape.
3. Map `GET /api/compliance/matters/{scenarioId}/timeline` in the existing endpoint module.
4. Return `404` for unsupported scenario ids instead of leaking an internal exception.
5. Add focused tests for:
   - happy-path timeline ordering and event coverage
   - delay-path blocker and clock-nearing-breach visibility
   - downstream-retry path showing failure, retry scheduling, and retried state
   - unsupported scenario id returns `404`
6. Update the execution board only to reflect `S2.4` progress.

## Acceptance Criteria

- one matter timeline endpoint exists and is backed by the replay harness
- the current decision path can be reconstructed from one inspectable thread
- delay and retry scenarios are understandable without reading raw infrastructure logs
- unsupported scenario ids return `404`
- the slice does not add queue views, dashboards, UI work, or a generalized event platform
