## Required Harness Capabilities

Implement:

1. one source-system stub shape for the safety intake signal
2. one downstream target stub shape with states:
   - `pending`
   - `acknowledged`
   - `failed`
   - `not required`
3. one configuration-backed clock model for the Germany affiliate follow-up window
4. deterministic replay for the three locked scenarios
5. scenario outputs written in operator-answer terms, not only raw transport terms

## Implementation Steps

1. Build the harness as a narrow internal API/service, not a generalized platform.
2. Keep all scenario data deterministic and code- or config-backed.
3. Add tests that prove replay determinism and expected operator-answer outputs.
4. Update the execution board only if needed to mark `S2.2` active or ready for review.

## Acceptance Criteria

- The outputs include enough human-answer terms to support the five operator questions later.
- The slice does not add a generic workflow engine, dashboard layer, or broad integration surface.
