## Interfaces Affected

Add one narrow internal-proof HTTP surface:

- `GET /api/compliance/matters/{scenarioId}/overview`

Use the locked replay scenario ids as the matter identifiers for this proof slice:

- `de-affiliate-happy-path`
- `de-affiliate-delay-path`
- `de-affiliate-downstream-retry`

Add one response contract that is explicitly matter-first and answers the operator's first-pass questions.

Required response fields:

- `scenarioId`
- `matterName`
- `jurisdiction`
- `owner`
- `activeClock`
- `blocker`
- `downstreamState`
- `downstreamAction`
- `decisionBasis`
- `clockName`
- `isClockActive`
- `isClockNearingBreach`
- `dueAtUtc`
- `timeRemaining`

## Preferred Implementation Shape

- keep the overview backed directly by `ComplianceMatterReplayHarness`
- add one narrow read-service method rather than a new generalized subsystem
- keep the route and contract in the existing compliance feature area
- update the JSON context only for the new overview contract

## Implementation Steps

1. Add a matter overview response type in the compliance contracts area.
2. Extend the read service with one method that maps replay output into the overview shape.
3. Map `GET /api/compliance/matters/{scenarioId}/overview` in the existing endpoint module.
4. Return `404` for unsupported scenario ids instead of leaking an internal exception.
5. Add focused tests for:
   - happy-path overview
   - delay-path overview with active clock
   - downstream-retry overview
   - unsupported scenario id returns `404`
6. Update the execution board only to reflect `S2.3` progress.

## Acceptance Criteria

- one matter overview endpoint exists and is backed by the replay harness
- the response is more useful for the five operator questions than the request-level read surfaces alone
- unsupported scenario ids return `404`
- the slice does not add a generalized workflow engine, queue view, timeline view, or UI work
